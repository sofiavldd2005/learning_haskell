# Creating my first project

Use `Cabal`. It is similar to Rust's `cargo`

```bash
cabal init -n --is-executable
```
> [!NOTE]
> `-n` skips the interactive questionnaire and just uses sensible defaults, while `--is-executable` tells Cabal we want to build a runnable program, not a library.

If you look at the generated files, you'll see a structure very similar to Rust:

`hello-haskell.cabal`: This is your Cargo.toml. It defines your project metadata, dependencies, and build targets.

`app/Main.hs`: This is your src/main.rs. It contains the entry point to your program.

--- 
##  Types

- A `Product Type` is just a collection of fields grouped together. In C/Rust, this is a `struct`.
In Haskell, you define the type name, the constructor name (usually the same as the type), and the types of its fields:
```haskell
-- Equivalent to: struct Sensor { char* name; int pin; };
data Sensor = Sensor String Int 
    deriving (Show) -- 'deriving (Show)' is exactly like Rust's '#[derive(Debug)]'
```

- If you want named fields (which is highly recommended for clarity), Haskell uses **Record Syntax**:

```haskell
data I2CDevice = I2CDevice {
    address   :: Int,
    connected :: Bool
} deriving (Show)

-- To instantiate it:
myDevice = I2CDevice { address = 0x68, connected = True }
```

- Sum Types (The enum / union)
A "Sum Type" means a value can be this OR that. In C, you'd use an unsafe union plus a tag. In Rust, it's an enum. In Haskell, you separate the variants using the pipe | character.

```haskell
-- A simple enum with no attached data
data PowerState = On | Off | Sleep 
    deriving (Show)

-- A more complex enum where variants hold distinct payloads (just like Rust!)
data SensorReading = 
      Temperature Float 
    | Humidity Float 
    | HardwareError String  -- Holds a string error message
    deriving (Show)
```

## Modules

 - I created the `src` directory, and the `Sensors` module
In Haskell, the module name must exactly match the file name (minus the .hs extension), and it must be capitalized. Let's move our types and functions over to this new file.


### Project structure ( as of now)

```
.
├── .git/                 # Your version control directory
├── .gitignore            # Ignores the build artifacts
├── hello-haskell.cabal   # Project configuration, dependencies, and build targets (Cargo.toml)
├── app/
│   └── Main.hs           # The impure entry point for your executable (src/main.rs)
├── src/
│   └── Sensors.hs        # Your pure library logic and data types 
└── dist-newstyle/        # Compiled binaries and object files generated by Cabal (target/)
```

> [!NOTE]
> A Quick Note on the Structure
> - app/ vs src/: This separation is a very common Haskell convention. The src/ directory is meant to be completely pure, reusable library code. The app/ directory acts as the "glue" that binds your pure library to the outside world (the IO monad, terminal input, file reading, etc.).
> -dist-newstyle/: If you haven't run cabal run recently, you might not see this folder. It is created automatically by Cabal the first time it builds the project, which is exactly why we added it to our .gitignore.
